# tui::i18n

## Purpose
English / Chinese string table for the TUI. Every user-facing string flows through here; choice of language is per-session, driven by locale or a hotkey.

## Public API
- `enum Lang { En, Zh }`.
- `struct T { lang: Lang }`.
- Lookup methods: `t.tab_skills()`, `t.help_normal_trash()`, `t.trash_groups_suffix(count)`, etc. — one method per string key or formatted sentence.

## Key invariants
- Every rendered string comes from here. Hardcoded strings in `ui.rs` are bugs.
- Adding a key: add to **both** languages in the same commit. A missing translation falls back to English (or fails to compile, depending on implementation choice).
- Trash UI labels (`tab_trash`, footer help, trash scope/status text) live here too — do not sneak English literals into `ui.rs` for new trash columns or help hints.
- Key names should describe intent, not content — `confirm_delete` not `are_you_sure_delete` — so copy changes don't require renames everywhere.

## Touch points
- **Upstream**: `tui::app` (owns the `T`), `tui::ui` (reads strings in draw functions).
- **Downstream**: none — this module is pure string lookup/formatting.

## Gotchas
- There is no locale auto-detection today; `tui::app::App::new` chooses the initial language and the user toggles with `l`.
- Don't concatenate translated fragments for sentences (`i18n.you_have() + n + i18n.items()`) — grammar rules differ between en/zh. Build full sentences with a format arg: `i18n.items_count_fmt(n)`.
- Destructive-action confirmation copy lives here for both languages. Keep the summary and impact text explicit about what is deleted versus only unlinked/removed from a group.
